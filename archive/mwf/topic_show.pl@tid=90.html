Topic Gosu / Gosu Exchange / Separate thread for game logic?
By ? Date 2009-02-26 05:09
I'm writing a game, and I want to have the game logic update at a different rate than the window.

This is fine when the window updates faster, but I noticed something strange when the window delay is slower than the logic thread's delay: the logic thread will wait for the window to update, even though they should be in separate threads.

Here's an example program: http://pastebin.com/f16e31a3d

If you set the window's delay to 16 (the default) then it prints out numbers at 20 fps, which you'd expect. Set it to 1000 and it prints numbers one per second.

What gives?
By jlnr (dev) Date 2009-02-26 10:11
Ruby's threading systems requires that you regularly call rb_schedule_thread (or was it rb_thread_schedule?) that will synchronously do the work of other threads. Gosu could be changed to call it periodically if you specify very large values for update_interval. Right now, Gosu::Window only calls this function as part of the update. Likewise, Gosu will also poll gamepads only once per update, so if you can work around the whole threading issue this would make things easier for sure ;)

What is your reasoning for redrawing only every 1000ms? Is the rendering so costly or is the app supposed to be as CPU efficient as possible?
By ? Date 2009-02-26 16:25
1000 was just to make the issue more noticeable. What I'm working towards is, if the computer is slow enough that it takes longer than the delay to draw a frame, I'd like it to drop frames (instead of slowing down the game logic).

It's not really a problem, I was just curious why it was happening.

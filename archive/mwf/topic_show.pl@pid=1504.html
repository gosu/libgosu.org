/**
*  This allows any key press to exit the logo screen and enter the game. It ignores mouse inputs
*  and keeps an eye on whether or not key presses are locked down.
*  @param button   The button pressed
*/
void BwkTitleMode::keyPressed( Gosu::Button button )
{
    if( _lockDown ) {
        return;
    }

    // Ignore the mouse
    if( button.getId() >= (int)Gosu::msRangeBegin && button.getId() <= (int)Gosu::msRangeEnd ) {
        return;
    }

    if( _curState != TITLE ) {
        _curState = (Screen_State)((int)_curState + 1);

        // Only show FMOD for windows distributions
#ifndef WIN32
        if( _curState == FMOD ) {
            _curState = (Screen_State)((int)_curState + 1);
        }
#endif
        _lockDown = true;
    }
}

/**
*  When any key is released that is not a mouse button, any lock down mode is turned off.
*  The mechanism of this lockdown between keyReleased() and keyPressed() keeps the user's
*  keypresses from triggering too quickly.
*  @param button The button pressed
*/
void BwkTitleMode::keyReleased( Gosu::Button button ) {
    // We exit if a mouse button was used so we can ignore those. Use the ID ranges to check.
    if( button.getId() >= (int)Gosu::msRangeBegin && button.getId() <= (int)Gosu::msRangeEnd ) {
        return;
    }

    _lockDown = false;
}

By jlnr (dev) Date 2009-09-12 17:15
Actually the boost::bind calls look okay. However it's a bit unusual to use it at all here (and I don't know why it didn't occur to me in the first place :)), it's more common to overload Window::buttonDown and Window::buttonUp to manually call methods on other objects. So if you want to go with the flow, you can maybe change your program this way and see if it helps ;)

Of course there is still no reason for this to crash, and I can only imagine that it would happen if the object you bind the callbacks to is destroyed in the meantime. In what kind of variable do you keep the object?
By jblovloss Date 2009-09-13 04:00
  I kind of tried something very odd here. I created these "mode" classes which take control of the update() and draw() calls. Essentially the main window just calls update of the current mode, draw of the current mode, and that's the end of its job. On each update, the current mode passes back a 'this' pointer to continue, or a pointer to a some new mode to switch over, making it dynamic... or NULL to exit. The TitleScreenMode is where this is happening, so the buttonup/down calls weren't there. But it just occured to me  - if you can overload Window::buttonUp, it should pass control of that to the new game mode as well. That fixes that problem, and lets me use it again in the future. Problem solved :D

  As to your question, said object is a singleton, so it never explodes.

By banister Date 2009-10-28 14:58 Edited 2009-10-28 15:02
Hi,

You're actually invoking some extremely expensive operations. Every frame you are creating a new image that is 500x500 pixels. You are also drawing a bezier curve (the most expensive graphical operation) across 500 pixels. Further, the bezier curve that is drawn must be sync'd again to opengl, the image that is sync'd is likely around 500x250 pixels. :)

There's a few things you can do to speed it up, though possibly not to real-time speeds. You could use the :sample_size hash_parameter (for bezier) to reduce the number of sample points. You should also, definitely, not create a new image every single frame, a better approach would be to create the image once and then use something like: image.rect 0, 0, @img.width, @img.height, :color => :black      to reset the image and then draw the bezier on the image.

I have a feeling this would not bring you to a particularly high frame-rate though.

If you are adamant to achieve a high frame-rate doing this kind of thing then I guess I can research faster ways to clear an image, though I have a feeling that this kind of application could never run at a very high frame rate in TexPlay.

A better approach may be to use raw OpenGL calls for this kind of thing? TexPlay is really designed for manipulating images --- whereas you seem to be wanting to draw a fresh curve at every frame?

EDIT: the user RunnerPack had something similar to this working at a high frame-rate, he used TexPlay's bezier function with a color_control proc and Gosu's built-in draw_line method to create a very nice real-time water effect. Maybe you can ask him to share his code :)

By banister Date 2009-10-28 16:12 Edited 2009-10-28 16:46
TexPlay has pretty good performance for its intended usage :) It is generally faster than ImageMagick for example and equal to or faster than a lot of other image manipulation frameworks. However you seem to be using TexPlay for another purpose than image manipulation.

Luckily TexPlay can also, sort of, be used for your kind of application too. Here is some code that may do what you want. It is using a combination of TexPlay (bezier math) and Gosu (draw_line):

require 'rubygems'
require 'gosu'
require 'texplay'
require 'time'

class W < Gosu::Window
    def initialize
        super(1024, 768, false)
        @img = TexPlay::create_blank_image(self, 500, 500)

        @frame_cnt = 0
        @beginning = Time.now
    end
   
    def draw
        @frame_cnt += 1

        passed_time = Time.now - @beginning
        puts @frame_cnt/passed_time
    end

    def update
        bez_array = []
        points = []

        (0..@img.width + 100).step(40) { |x|
            p = TexPlay::TPPoint.new
            p.x = x
            p.y = @img.height * rand

            points << p
        }

        p = points.shift
        @img.bezier points, :sample_size => 40, :color_control => proc {  |c,x,y|
            draw_line(
                      p.x + 100, p.y + 100, 0xffaabbcc,
                      x + 100, y+ 100, 0xffaabbcc
                      )
            p.x = x
            p.y = y
            :none
        }
    end

end

w = W.new
w.show

The above code runs reasonably fast on my system, it at least has a double digit frame rate :)

EDIT: just to make things a bit clearer, i really dont think your code would run any faster even if it was written in pure C . Unfortunately it's just the nature of image manipulation in OpenGL --- i.e OpenGL does not support it ;) If you want to manipulate an OpenGL texture the following things must happen: (1) you must first modify a local copy of the opengl texture (2) you must resync that local copy to OpenGL. 

It's really number (2) that is the very expensive operation, and if you're resyncing a reasonably large texture back to opengl at every frame you will not, unfortunately get a good frame rate. :(
TexPlay tries its best by only syncing that part of the texture that has changed but still it can be quite slow if you are changing a large portion of a large texture. In practical terms, though, I can't imagine large scale manipulation of opengl textures to happen in a real game. I wrote a worms-style game prototype that does alot of image manipulation, i.e blowing holes in the landscape etcetara and that game was able to maintain around 50 FPS. I can't think of any real game where you need to modify large chunks of large images every frame?

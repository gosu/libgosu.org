I'm designing the first game I've ever made (remaking Sokoban) and I'm having trouble figuring out how to divide the game into different levels. Like a lot of people new to Gosu probably did, I just started modifying the Starfighter tutorial game. I have all of my game objects declared in the Window initialize method and I'm not sure if they can (or should) be declared anywhere else. I looked at the ruby main loop and saw that the only things that update are game input, the update method, and the draw method. The problem I'm having is I don't see any way to create new objects for new levels if the initialize method is where they are and it is never updated. By jlnr (dev) Date 2009-06-19 14:57
The only requirement coming from Gosu is that any drawing operations (image.draw, draw_quad etc.) have to take place while Window#draw is running, i.e. you cannot draw anything in direct response to a button_down. Other than that, you are free to do whatever the Ruby language supports, and that means creating things everywhere :)

A very basic approach would be to have a start_level method that clears the object arrays etc., and is called by the Window constructor and every time that a level is finished.

However, this gets ugly as soon as you have several different screens like the main menu, the actual game, a help screen etc. The usual approach to that is to create classes like GameState that have the same callbacks as Window does (draw, update, button_up, button_down). The Window then has a reference to the current game state and forwards all its methods. An example of this with some extra features can be found here: http://www.libgosu.org/cgi-bin/mwf/topic_show.pl?tid=113
